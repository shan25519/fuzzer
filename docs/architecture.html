Protocol Fuzzer Architecture

A deep dive into the design, flows, and engines of the multi-protocol fuzzing suite.

1. System Overview

The tool is designed as a modular fuzzer that abstracts protocol-specific complexities into specialized "Engines". A single scenario can be executed against various transport layers (TLS over TCP, TLS over QUIC, or raw HTTP/2 streams).

2. Test Execution Flow

Every test run follows a strict Plan -> Act -> Validate cycle. The fuzzer compares the target's behavior against a "Well-behaved" baseline (OpenSSL or Node.js native) to determine if a protocol violation was handled correctly.

3. QUIC Implementation Matrix

QUIC is unique because it requires complex state management. The fuzzer uses three distinct paths based on the testing goal:

graph LR subgraph Stateless A[Custom Packet Builder] -->|Hand-crafted| B(Initial Fuzzing) B -->|Focus| C[Handshake Parsers] end subgraph Stateful D[OpenSSL 3.6 Wrapper] -->|s_client| E(1-RTT Established) E -->|Focus| F[HTTP/3 & App Layer] end subgraph Native G[Quiche N-API] -->|Native Binary| H(Full State Machine) H -->|Focus| I[Deep Protocol Fuzzing] end

QUIC Stateless: Best for raw byte mutations in Initial packets.
OpenSSL: Best for testing application-layer resilience once the pipe is encrypted.
Quiche: The high-performance, native path for full protocol compliance.

4. PAN-PQC SNI Evasion Logic

This advanced suite tests whether firewalls can correctly categorize URLs when the SNI is hidden or split across packets using large Post-Quantum Cryptography (PQC) payloads.

5. Distributed Architecture

For large-scale infrastructure testing, the fuzzer can deploy agents across multiple regions to probe targets from diverse network paths.
